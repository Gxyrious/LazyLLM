# IR 实验整体架构

## 一、核心目标

`ir_experiment` 用于比较 JSON、XML、Markdown 三种模型输入表示对文档修改效果的影响。三条路线共享同一个核心 `WriterDocument`、同一个 `ModifyPlan`、同一个 `PatchSet` 和同一套 Apply / 飞书写回逻辑。

实验变量只是 LLM 看到的文档表示：

```text
WriterDocument
    ├── StructuredJsonAdapter → JSON
    ├── XmlAdapter            → XML
    └── MarkdownAdapter       → Markdown
                                 │
                                 ▼
                         LLM → ModifyPlan → PatchSet
                                 │
                                 ▼
                     apply_patch(WriterDocument, PatchSet)
                                 │
                                 ▼
                       本地结果 / 飞书写回
```

XML 和 Markdown 投影不是新的核心 IR，也不定义路线专属 Patch。`PatchSet` 始终直接定位 `WriterBlock.node_id` 并应用到原始 `WriterDocument`。

## 二、文件职责

```text
ir_experiment/
├── writer_ir.py             # 核心 DocIR、ModifyPlan 和统一 Patch 类型
├── xml_projection.py        # WriterDocument ↔ XML 可编辑语义投影
├── markdown_projection.py   # WriterDocument ↔ Markdown 可编辑语义投影
├── model_adapters.py        # 三种模型输入和 Patch 边界校验
├── experiment_tools.py      # read → plan → patch → apply → writeback
├── test.py                  # 投影往返验证和真实 LLM 端到端入口
└── docs/
    ├── 01-整体架构方案.md
    ├── 03-xml和md结构设计.md
    └── 04-Patch重构实施方案.md
```

`writer_ir.py` 是唯一的修改语义定义处。投影文件只负责可编辑语义的等价表示，不承担 Patch 执行。

## 三、核心类型

### 3.1 文档 IR

- `WriterDocument`：文档 ID、标题、阶段和根 Block。
- `WriterBlock`：稳定 `node_id`、块类型、文本、Span、子 Block 和必要语义属性。
- `WriterSpan`：连续文本及其行内样式。

### 3.2 修改计划

`ModifyPlan` 用于让 LLM 先完成定位和操作拆分。每个 `ModifyInstruction` 对应一个现有 Block 的一次 `replace / insert / delete / move` 操作。

`insert` 直接使用 `target_node_id` 作为插入锚点；`move` 额外使用 `anchor_node_id`。不再定义 `Anchor` 类型。

### 3.3 统一 Patch

```text
PatchSet
    └── PatchHunk[]
            └── PatchBlock[]   # 仅 insert 使用
```

`PatchHunk` 保留一个类型，用 `modify_type` 区分四种操作：

- `replace`：`target_node_id + text_range + new_text`；
- `insert`：`target_node_id + position + new_blocks`；
- `delete`：`target_node_id`；
- `move`：`target_node_id + anchor_node_id + position`。

`old_text` 不由 LLM 生成。Adapter 在解析模型结果后从原始 `WriterDocument` 填充，供本地 Apply 和远端写回做并发一致性校验。

`PatchBlock` 只表示当前可新建的简单 Block：

```text
type + content + numbering
```

新 Block ID 由 Apply 生成。当前不在 `PatchBlock` 中生成嵌套子树或复杂富文本。

## 四、Adapter 边界

### 4.1 模型输入

- JSON：序列化 `WriterDocument` 必要字段，作为基线。
- XML：用原生 XML 标签表达 Block 和行内样式，`id` 对应 `node_id`。
- Markdown：用 Markdown 原生语法表达标题、列表和样式，用 `<!-- id="..." -->` 标记 Block 边界。

### 4.2 模型输出

三条路线都通过同一个 Pydantic `PatchSet` schema 请求结构化输出。Adapter 只做以下确定性工作：

1. 校验目标文档和 Block ID；
2. 校验 `text_range` 和移动锚点；
3. 拒绝模型提供的 `old_text`；
4. 从原文档填充 `replace / delete` 的 `old_text`；
5. 标记当前模型表示路线。

Adapter 不修正 LLM 选择的目标和操作，不从 XML / Markdown 表面文本计算 Patch。

Patch 生成阶段会为 Plan 中的 replace 目标提供一份三条路线完全相同的字符索引表。它只明确 `WriterBlock.content` 的 Unicode 偏移，避免让 LLM 凭空计算中文字符数；目标 Block 和待替换语义仍由 LLM 决定。

---

## 五、Apply 与写回

`apply_patch` 始终在文档深拷贝上执行。任一 hunk 校验或执行失败时，返回未修改的文档和失败 `PatchResult`。

Apply 只执行 Patch 明确声明的操作：

- 按字符范围替换局部文本；
- 保留范围外的 Span；
- 插入、删除或移动一个 Block；
- 同一 Block 的多个 replace 按原文范围逆序执行。

Apply 不推导章节边界、不自动修改编号、不基于全文 diff 猜测局部变化。

飞书写回只在本地 Apply 成功后开始。写回前重新读取远端 Block 并预校验文本、父子关系和当前接口能力；写回逻辑不另行推导修改。

---

## 六、当前边界

当前优先跑通“单种格式 + 单次任务”，暂不实现：

- 任务 9 的长文档上下文裁剪；
- 任务 10 的 Outline → Draft → Final 多阶段生成；
- URL 与平台原生交叉引用；
- stage 修改；
- 新建复杂嵌套 Block 和新建复杂富文本。

任务 1–8 通过局部 replace、Block insert/delete/move 及显式编号替换表达。如果一个逻辑章节在 `WriterDocument` 中由多个平级 Block 组成，LLM 必须为每个 Block 输出独立 hunk。
