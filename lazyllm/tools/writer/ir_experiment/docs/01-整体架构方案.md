# IR 实验整体架构方案

本方案重构 `lazyllm/tools/writer/ir_experiment` 目录，把现有 Structured JSON 纵贯线从"混用 `data_models` 旧 IR"迁移到"内层 `WriterDocument` 唯一、外层多路线投影"的清晰两层架构，并搭好 JSON / XML / Markdown 三条实验路线的骨架。本次落地范围限定为**方案甲**：`replace / insert / delete` 三种 patch 操作，原生 `move` 与编号重排留到后续。

---

## 一、两层架构

```
内层 IR（唯一，永不直接进 LLM 上下文）
    WriterDocument / WriterBlock / WriterSpan          ← writer_ir.py
    PatchHunk / PatchSet / ModifyPlan / PatchResult     ← writer_ir.py（统一 patch 语义）
        │
        │  Adapter.document_to_model_input(WriterDocument) -> 模型可见的字符串
        ▼
外层表示（实验变量，每条路线独立）
    JSON   路线：to_prompt_json(WriterDocument)           ← StructuredJsonAdapter
    XML    路线：树形 <block> 投影（占位）                  ← XmlAdapter
    Markdown 路线：带锚点的 Markdown 投影（占位）           ← MarkdownAdapter
        │
        │  LLM 输出：每条路线自己的 Plan / Diff 表示
        ▼
    Adapter.model_output_to_patch(output, WriterDocument) -> PatchSet
        │
        ▼
统一 Patch（外层，但面向内层 WriterBlock.node_id）
    （JSON 路线恒等映射，无独立外层表示）
        │
        │  apply_patch 作用回内层
        ▼
内层 IR（已修改的 WriterDocument）
        │
        ▼
平台写回（飞书 / Notion）
```

核心约束（对应 README 五节立论）：

- 内层 `WriterDocument` 唯一，三条路线**不创建各自的 Document IR**。
- 三条路线的差异**只存在于 Adapter**（以及未来各自路线的 diff 中间表示）；统一 patch 语义、`read / apply / writeback` 三段全部共用。
- 外层 `PatchSet` 统一面向 `WriterBlock.node_id`，三条路线产出的 patch 走同一个 `apply_patch`。

## 三、各文件的具体改动

### 3.1 `writer_ir.py`（新增 patch 类，内层 IR 不动）

本次把**统一 patch 语义**并入本文件。理由：patch 操作面向 `WriterBlock.node_id`，与内层 IR 同属一个语义域，且三条路线共用同一套 `apply_patch`，放任何单一路线的文件里都名不副实。JSON 路线是恒等映射，不存在 JSON 专属的中间表示，因此也不需要 `json_projection.py`。

内层 IR 四个类（`WriterDocument / WriterBlock / WriterSpan / WriterStage`）保持现状不改。新增的 patch 类：

- `Anchor`：`node_id: str` + `heading_path: list[str]` + `text_offset: int|None`。定位语义为"在某 node 内的文本偏移"，跨平台稳定。
- `ModifyInstruction`：`target_node_id: str` + `modify_type: Literal['replace','insert','delete']` + `position: Literal['before','after']|None`（仅 insert 用） + `instruction: str`。
- `ModifyPlan`：`plan_id / task_id / scope: Literal['document','section','block','span'] / target_node_ids: list[str] / instructions: list[ModifyInstruction] / summary / meta`。
- `PatchHunk`：`target_node_id / modify_type / anchor: Anchor|None / old_text: str|None / new_text: str|None / position: Literal['before','after']|None / meta`。应用阶段按列表顺序生成内部 hunk 标识，不依赖模型生成 ID。
- `PatchSet`：`patch_id / target_doc_id: str / hunks: list[PatchHunk] / meta`。
- `PatchResult`：`patch_id / success: bool / applied_hunks: list[str] / failed_hunks: list[str] / message / meta`。

`modify_type` 集合即方案甲的 `replace / insert / delete`，不含原生 `move`。

### 3.2 `model_adapters.py`（改）

统一 Adapter 接口：

```python
class ModelAdapter(Protocol):
    name: str
    def document_to_model_input(self, document: WriterDocument) -> str: ...
    def model_output_to_patch(self, model_output: Any, document: WriterDocument) -> PatchSet: ...
```

三条路线：

- `StructuredJsonAdapter`（实装）：`document_to_model_input` = `to_prompt_json(document)`，保持恒等映射；`model_output_to_patch` 校验 `PatchSet`，比对 `patch.target_doc_id == document.document_id`，写入 `meta['model_representation'] = 'json'`。
- `XmlAdapter`（占位）：两个方法都 `raise NotImplementedError('XML projection not implemented yet')`。
- `MarkdownAdapter`（占位）：保持现状两个方法 `NotImplementedError`。

三条路线 Adapter 的 `name` 分别为 `'json' / 'xml' / 'markdown'`，供 `apply_patch` 与写回侧标注来源。

### 3.3 `experiment_tools.py`（改，主体）

导入切换：

- 删除 `from ..data_models.docir import ...` / `from ..data_models.revision import ...`。
- 改为 `from .writer_ir import WriterDocument, WriterBlock, WriterSpan, ModifyPlan, PatchSet, PatchHunk, PatchResult`（patch 类与内层 IR 同源）。
- `WritingTask / TargetDocument` 继续 `from ..data_models.task import`（任务描述，非 IR）。

`__public_apis__` 改为（route 三对 + apply + writeback）：

```python
__public_apis__ = [
    'read_writer_doc',
    'generate_modify_plan_json', 'generate_modify_plan_xml', 'generate_modify_plan_markdown',
    'generate_patch_json',        'generate_patch_xml',        'generate_patch_markdown',
    'apply_patch',
    'write_patch_to_feishu',
]
```

构造器持三个 Adapter 实例：`self.json_adapter / self.xml_adapter / self.markdown_adapter`。

纵贯线五段：

1. **读**：`read_writer_doc(feishu_url) -> WriterDocument`。`_resolve_feishu` 不变；`_build_block_tree` → `_build_writer_blocks`，按 `parent_id` 重建 `WriterBlock` 树；`_to_doc_block` → `_to_writer_block`，填 `node_id / type / content / stage='final'`，level 写入 `numbering={'level': n}`，飞书原始 id/type/parent 暂存 `source_refs`。
2. **Plan**：`_generate_modify_plan(adapter, task, document)` 为三条路线共用私有方法；`generate_modify_plan_json/xml/markdown` 只差传哪个 adapter。Plan prompt 文案把 `block_id` 统一改为 `node_id`，校验 `node_id` 存在。
3. **Patch**：`_generate_patch(adapter, modify_plan, document)` 共用；`generate_patch_json/xml/markdown` 分路由。JSON 路线 `_call_llm_structured(prompt, PatchSet)`；XML/Markdown 路线在 Adapter 内 `NotImplementedError`。
4. **Apply**：`apply_patch(document, patch_set) -> (WriterDocument, PatchResult)` 实现 replace / insert / delete：
   - replace：`target.content = hunk.new_text`，清空 `target.spans`。
   - insert：按 `hunk.target_node_id` 找锚点 block，按 `hunk.position`（before/after）在其父的 children 列表里插入新 `WriterBlock`（`node_id` 由 patch 指定或生成）。
   - delete：从父的 children 列表移除该 node。
   - 其余 `modify_type` 直接进 `failed_hunks`，快速失败，不静默忽略（AGENTS.md 第 4 条）。
   - 应用后重算 `plain_text` 等价缓存（若有）。
5. **写回**：`_write_patch_to_feishu` 改动最小，`target_id` 取值从旧 `block_id` 字段名切到 `node_id`；其余飞书调用不变。

### 3.4 `test.py`（改）

- `_build_task` 继续用 `data_models.task.WritingTask`。
- `run_structure_experiment` → `run_json_experiment`，入口调 `generate_modify_plan_json / generate_patch_json`。
- 新增 `run_xml_experiment / run_markdown_experiment` 占位（`NotImplementedError`），与 Adapter 占位对称。
- `test_structure_live` → `test_json_live`，skip 条件不变（仍需真实飞书 + Qwen 凭证，AGENTS.md 第 10 条禁止 mock LLM）。

## 四、不做的事（方案甲边界）

明确划在本次范围外，避免 scope creep：

- 原生 `move` patch 操作（README 任务 5）。方案甲阶段用 delete + insert 组合表达，等价但不优雅。
- 自动编号重排（README 任务 6）。编号重排依赖写回侧平台语义，放到方案乙。
- Markdown / XML 路线的真正实装。本次只搭 Adapter 占位与三对入口。
- 公共测试文档与评估指标采集脚手架（README 七节）。那是"跑实验"阶段的事，本次只搭"能跑实验的结构"。

---

## 五、迁移后与方案乙的衔接

方案甲落地后，向方案乙（加 `move` + 编号重排）扩展的改动面收敛在：

- `writer_ir.py`：`PatchHunk.modify_type` 加 `'move'`，新增 `position` 的 `inside` 语义或独立 `target_parent_node_id`。
- `experiment_tools.apply_patch`：加 move 分支（子树摘除 + 重插）。
- 编号重排：在 apply 后加一步 `_renumber(document)`，按 `numbering.level` 重算有序编号，集中在内层 IR 操作，不污染 patch 语义。

XML / Markdown 路线的实装则各自独立：新建 `xml_projection.py` / `markdown_projection.py`，在对应 Adapter 里完成"投影 + diff 编译"，`experiment_tools.py` 完全不用改（因为 Plan/Patch 入口已经是三路由对称结构）。
