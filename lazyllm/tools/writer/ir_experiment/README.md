## 五、三种实验策略

### JSON 投影

- 将 `WriterDocument` 的必要字段直接序列化给模型。
- 作为低转换损失的基线，验证模型是否能直接理解结构化 Block IR。
- 不需要为此创建“JSON IR”。

### XML 投影

- 参考飞书官方 CLI 的思路，把树形层级、Block 类型、ID 和必要样式投影为对模型更友好的 XML。
- 重点验证复杂层级、移动、插入和多个重复文本场景下的定位能力。
- XML 不是系统内的第二套 IR。

### Markdown 投影

- 把同一份 `WriterDocument` 投影为模型熟悉的 Markdown，但要设计可定位的 Block 锚点和不干扰正文理解的标记。
- 模型不输出 Git diff，而是直接输出与 JSON、XML 路线相同的统一 `PatchSet`。
- Markdown Adapter 只负责投影和统一 Patch 边界校验，不创建 `MarkdownDocument` 或 `MarkdownDocIR`。

## 六、实验目标

主问题是：在使用同一份原文档、同一个修改任务、同一模型和同一采样参数时，JSON、XML、Markdown 哪种投影能够：

1. 更准确地理解文档结构和用户要求。
2. 更准确地定位目标 Block 和目标文本范围。
3. 更正确地生成 Block 增、删、改、移动 Patch。
4. 在不误改无关区域的前提下产生更好的写作文本。
5. 写回飞书/Notion 后更好地保持段落边界、Block 顺序、层级、编号、引用和样式。
6. 在长文档中使用更少的 token 和可接受的时延完成同一任务。

## 七、实验方案

### 公共输入

- 准备一份共用的飞书测试文档，覆盖标题层级、普通段落、粗体/斜体/链接、有序和无序列表、嵌套、表格、自动编号和交叉引用。
- 每条路线都从这份文档的同一个快照读取为 `WriterDocument`。
- 写回验证时为 JSON、XML、Markdown 使用三份独立副本，避免前一条路线污染后一条路线的输入。
- 三条路线使用相同模型、system prompt 主体、任务描述、采样参数和重试次数。只允许格式说明和输出 schema 随投影策略变化。

### 任务集

1. 唯一文本的局部替换。
2. 文档中存在重复文本，必须依赖标题路径或 Block ID 定位。
3. 在指定 Block 前/后插入新段落。
4. 删除 Block，不损坏前后兄弟顺序。
5. 移动 Block 或整个子树。
6. 增删或移动章节后重排自动编号。
7. 修改被引用的章节，保持交叉引用有效。
8. 修改部分富文本，保持未修改文字的样式。
9. 长文档中仅向模型提供目录和相关片段。
10. 从空文档生成 Outline，继续扩写为 Draft 并完成 Final，验证三阶段始终使用同一 IR。

### 评估指标

- 目标 Block 定位准确率。
- 操作类型和插入/移动位置准确率。
- 模型输出可解析率和 Patch 可应用率。
- 文本质量，使用盲评或固定 rubric 评估需求遵循、事实性、连贯性和风格一致性。
- 非目标区域意外改动数量。
- Block 数量、顺序、层级和段落边界保持率。
- 自动编号和交叉引用正确率。
- 富文本样式保持率。
- 写回后重新读取与预期 `WriterDocument` 的一致性。
- token 消耗、时延和重试次数。

### 看结果的顺序

1. 先判断模型是否选中正确目标并生成正确 Patch。
2. 再判断 Patch 在本地应用后的 IR 是否正确。
3. 最后判断写回飞书并重新读取后是否保持结构和样式。

这样可以区分“模型理解/生成问题”、“Patch 应用问题”和“平台 Adapter 写回问题”，避免把三类失败混成一个结果。
